## 1) Create a proxy

Create a proxy in Xenovia with:

- `owner`
- `environment`
- `declared_scope` (high-level allowed actions)

<Callout type="info">
  Proxies are created in the platform. Runtime agents and runtime tools are derived from requests that hit the runtime proxy.
</Callout>

## 2) Route runtime traffic and discover metadata

Point your runtime repo to the Xenovia runtime proxy and execute a known workflow. As traffic arrives, Xenovia registers runtime-derived agent/tool identities under the selected proxy for tracing and policy evaluation.

<Callout type="warn">
  Avoid broad wildcard permissions during initial rollout. Start narrow and expand using observed runtime behavior.
</Callout>

## 3) Add a runtime policy

Define a baseline policy:

- Allow low-risk reads in scope.
- Require approval for write actions crossing environment boundaries.
- Block actions outside declared scope.

<Tabs>
  <Tab title="Allow">
    Safe, in-scope read operations with low blast radius.
  </Tab>
  <Tab title="Escalate">
    Sensitive writes or cross-boundary changes requiring human approval.
  </Tab>
  <Tab title="Block">
    Out-of-scope or explicitly restricted actions.
  </Tab>
</Tabs>

## 4) Run and inspect

Execute a known workflow and inspect:

- Proxy + runtime-derived identity.
- Requested action and inferred intent.
- Scope and policy comparison.
- Final runtime decision.
- Linked trace evidence.

## 5) Validate remediation flow

Trigger a controlled out-of-scope request and confirm operators can:

- Review context.
- Approve or deny with reason.
- Update policy without losing historical evidence.
