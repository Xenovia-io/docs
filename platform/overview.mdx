## One surface across runtime operations

Xenovia keeps proxies, runtime-derived agents/tools, policies, traces, and remediation in one operational surface so teams can reason from shared context instead of fragmented logs.

<CardGroup cols={3}>
  <Card title="Proxies" icon="hexagon-nodes">Operator-created inventory, owners, and live runtime state.</Card>
  <Card title="Runtime Identities" icon="fingerprint">Agents and tools derived from runtime proxy traffic.</Card>
  <Card title="Policies" icon="gavel">Enforcement logic with approval routing.</Card>
  <Card title="Traces" icon="file-waveform">Evidence from request to outcome.</Card>
  <Card title="Remediation" icon="screwdriver-wrench">Guided operator response workflows.</Card>
  <Card title="Governance" icon="shield">Shared context across security and ops teams.</Card>
</CardGroup>

## Core entities

- **Proxies**: Platform-created runtime entry points with ownership and scope.
- **Runtime agents/tools**: Derived from runtime traffic under each proxy.
- **Policies**: Rules, approvals, and escalation thresholds.
- **Traces**: Decision evidence and execution history.
- **Operators**: Reviewers responsible for approvals and incident response.

## Runtime flow

<Steps>
  <Step title="Action request">Runtime call reaches a Xenovia proxy.</Step>
  <Step title="Identity mapping">Runtime-derived agent/tool identity is associated to that proxy.</Step>
  <Step title="Intent reasoning">Xenovia interprets requested intent and risk.</Step>
  <Step title="Scope comparison">Request is checked against allowed boundaries.</Step>
  <Step title="Policy decision">Action is allowed, blocked, or escalated.</Step>
  <Step title="Evidence and response">Trace is stored and operator workflow is triggered when needed.</Step>
</Steps>
