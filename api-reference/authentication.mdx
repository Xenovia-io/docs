All API requests must include a bearer token in the `Authorization` header:

```bash
Authorization: Bearer <your-token>
```

<Callout type="warn">
  Do not send API tokens from browser clients. Keep all privileged requests server-side.
</Callout>

## Token handling

- Issue tokens through your Xenovia workspace or approved service integration.
- Keep tokens server-side and never expose them in client-side bundles.
- Rotate tokens on a scheduled cadence and after role changes.

## Service-to-service recommendation

For backend integrations:

- Store tokens in a secure secret manager.
- Attach tokens at request time.
- Scope each token to the smallest required API surface.

## Environment separation

Use different credentials for each environment:

- `dev`
- `staging`
- `production`

Never reuse production credentials in non-production pipelines.

## Recommended auth flow

<Steps>
  <Step title="Issue">Create scoped credentials for one environment.</Step>
  <Step title="Store">Save token in your secrets manager.</Step>
  <Step title="Inject">Attach token server-side at request time.</Step>
  <Step title="Rotate">Rotate on schedule and after role changes.</Step>
</Steps>

## Authentication failures

If authentication fails, the API returns `401 Unauthorized` or `403 Forbidden` depending on token validity and access scope.
